## Назначение

F2FS Guardian контролирует реальный экземпляр F2FS, на котором расположен `/data`, и может запросить короткую штатную сессию сборки мусора ядра только при высокой заполненности файловой системы и нахождении телефона в подтверждённом безопасном окне.

Это не CPU/GPU-твик, не модуль повышения FPS, не способ подмены результатов тестов UFS и не универсальный «оптимизатор». Модуль не может увеличить физическую максимальную скорость накопителя. Его узкая задача — переносить ресурсоёмкое обслуживание F2FS на время простоя и делать поведение системы более предсказуемым при сокращении свободного пространства.

## Статус релиза

**v1.1 — текущий стабильный релиз автора Lolokeksu.**

Версия v1.1 сохраняет консервативную политику F2FS из v1 и добавляет улучшения совместимости и интерфейса. Режимы GC ядра, защитные пороги и правила владения не изменены.

Аппаратная интеграция проверена на:

| Устройство | Android | Root | `/data` | Результат |
|---|---:|---|---|---|
| Realme GT Neo 5 SE | 13 / API 33 | APatch | F2FS, `dm-51` | Установщик, self-test, демон, статус, очередь и отмена подтверждены |

Реальная сессия GC намеренно не форсировалась во время проверки, поскольку накопитель был заполнен только на 19%, а защитная политика корректно сообщила об отсутствии триггера. Это ожидаемое поведение, а не неполная установка.

Изменения совместимости и интерфейса v1.1 прошли статические и mock-тесты. После установки в KernelSU Next, SukiSU и других специфичных средах всё равно необходимо выполнить `self-test`.

## Что нового в v1.1

- добавлен каскадный детектор файловой системы `/data` для KernelSU Next и других изолированных сред установщика;
- отсутствие результата в установщике больше не считается доказательством несовместимости `/data`;
- runtime-проверка остаётся строгой и блокирует запись в ядро, пока F2FS и все обязательные узлы не подтверждены;
- добавлено точное сопоставление `major:minor` с блочным устройством и экземпляром F2FS;
- удалён небезопасный запасной выбор первого найденного экземпляра F2FS;
- добавлено двуязычное русско-английское терминальное меню с сохранением выбранного языка;
- добавлены короткие команды `f2g`, `f2status`, `f2doctor`, `f2start`, `f2stop` и другие;
- добавлена расширенная диагностическая команда `doctor`;
- перед запуском демона добавлено ожидание завершения загрузки Android;
- исправлен бесконечный цикл `Invalid selection` в окнах Manager Action без интерактивного stdin;
- кнопка Action теперь показывает панель состояния и предлагает использовать `f2g` в Termux или ADB для интерактивного управления.

## Почему такая архитектура

F2FS уже содержит собственный сборщик мусора. F2FS Guardian не заменяет его. Модуль добавляет консервативный слой управления стандартным sysfs-интерфейсом `gc_urgent`:

1. определяет точный экземпляр F2FS, обслуживающий `/data`;
2. считывает заполненность файловой системы и статистику сегментов;
3. проверяет состояние экрана, зарядку, батарею, температуру и блочную I/O-нагрузку;
4. запрашивает только документированный режим GC F2FS;
5. применяет строгий лимит времени;
6. возвращает принадлежащий модулю режим в `gc_urgent=0`;
7. немедленно останавливается при изменении защитных условий или переходе управления к другому инструменту.

## Совместимость

### Требуется

- Android 13, 14, 15 или 16 — API 33–36;
- Magisk 20.4+, KernelSU, KernelSU Next, совместимые с KernelSU среды SukiSU или APatch;
- раздел `/data`, отформатированный в F2FS;
- доступный для записи `/sys/fs/f2fs/<instance>/gc_urgent`;
- доступные для чтения `free_segments` и `dirty_segments` для автоматического режима;
- доступная статистика блочного устройства для проверки низкой I/O-нагрузки.

### Не требуется

- Zygisk;
- LSPosed;
- Termux для фоновой работы;
- отдельный модуль BusyBox;
- кастомное ядро, если штатное ядро предоставляет необходимые узлы F2FS.

Версия Android сама по себе не подтверждает совместимость. Производитель может удалить, переименовать или ограничить интерфейсы F2FS в ядре. После каждой смены прошивки или ядра выполняйте `self-test`.

## Гарантии безопасной логики

F2FS Guardian v1.1:

- использует только `gc_urgent=2` для обычной работы и `gc_urgent=1` для короткой критической сессии;
- никогда не записывает недокументированный режим `4`;
- никогда не меняет `gc_urgent_sleep_time`;
- никогда не меняет права файлов в `/sys`;
- отказывается перезаписывать ненулевое значение `gc_urgent`, установленное другим инструментом;
- повторно проверяет безопасность во время активной сессии;
- сбрасывает только тот режим, владение которым может подтвердить;
- хранит конфигурацию как проверенные целочисленные данные и никогда не выполняет её через `source` или `eval`;
- не содержит нативных исполняемых файлов, загрузчиков, socket-клиентов, телеметрии или удалённого кода;
- не изменяет SELinux, AVB, dm-verity, `boot`, `init_boot`, `vendor_boot`, `dtbo`, `vbmeta` или динамические разделы;
- не использует установочное меню на кнопках громкости.

## Сбалансированная политика по умолчанию

Перед автоматическим или поставленным в очередь обслуживанием должны выполняться все защитные условия.

| Условие | Значение по умолчанию |
|---|---:|
| Интервал проверки | 60 минут |
| Минимальный интервал между успешными сессиями | 24 часа |
| Время с выключенным экраном | минимум 20 минут |
| Требование зарядки | да |
| Минимальный заряд батареи | 50% |
| Максимальная температура батареи | 39,0 °C |
| Максимальная наблюдаемая блочная I/O-активность | 25 операций/с |

Обычное обслуживание требует выполнения обоих условий:

| Обычный триггер | Значение по умолчанию |
|---|---:|
| Заполнение `/data` | не менее 84% |
| Dirty-сегменты | не менее 256 |
| Режим ядра | `gc_urgent=2` |
| Максимальная длительность | 480 секунд |

Критическое обслуживание требует выполнения всех трёх условий:

| Критический триггер | Значение по умолчанию |
|---|---:|
| Заполнение `/data` | не менее 95% |
| Dirty-сегменты | не менее 128 |
| Свободные сегменты | не более 96 |
| Режим ядра | `gc_urgent=1` |
| Максимальная длительность | 90 секунд |

Это консервативные значения политики модуля, а не универсальные константы F2FS. Их можно изменить.

## Установка

1. Создайте резервную копию важных пользовательских данных.
2. Скачайте `F2FS-Guardian-v1.1.zip` со страницы GitHub Releases.
3. Установите его через Magisk, KernelSU или APatch Manager.
4. Перезагрузите устройство.
5. Выполните самопроверку:

```sh
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh self-test
```

Ожидаемый успешный результат:

```text
PASS: runtime prerequisites are available
```

6. Проверьте состояние:

```sh
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh status
```

Результат `no trigger` при низкой заполненности накопителя является правильным. Не форсируйте сессию GC только ради доказательства возможности записи в ядро.

## Команды

Канонический CLI остаётся доступен:

```sh
# Полный статус
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh status

# Оценить триггер и защитные условия без запуска обслуживания
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh check

# Поставить безопасное обслуживание в очередь; оно всё равно ожидает выполнения всех условий
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh request

# Отменить запрос в очереди или остановить принадлежащую модулю активную сессию
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh cancel

# Показать последние записи журнала
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh logs

# Показать постоянную конфигурацию
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh config

# Открыть двуязычное терминальное меню
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh menu

# Применить профиль
su -c '/data/adb/modules/f2fs_guardian/f2fs-guardian.sh profile balanced'
su -c '/data/adb/modules/f2fs_guardian/f2fs-guardian.sh profile conservative'
su -c '/data/adb/modules/f2fs_guardian/f2fs-guardian.sh profile manual'

# Выбрать язык интерфейса
su -c '/data/adb/modules/f2fs_guardian/f2fs-guardian.sh lang en'
su -c '/data/adb/modules/f2fs_guardian/f2fs-guardian.sh lang ru'

# Включить или отключить автоматическое обслуживание
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh enable
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh disable

# Расширенная диагностика совместимости
su -c /data/adb/modules/f2fs_guardian/f2fs-guardian.sh doctor
```

После перезагрузки systemless-обёртки предоставляют короткий интерфейс:

```sh
f2g
f2status
f2check
f2request
f2cancel
f2logs
f2doctor
f2start
f2stop
f2profile balanced
f2profile conservative
f2profile manual
f2lang en
f2lang ru
```

`f2start` включает автоматическое обслуживание, но не запускает принудительную сессию GC немедленно. `f2request` ставит ограниченный запрос в очередь и по-прежнему соблюдает все защитные условия.

Termux необязателен для фоновой работы. Он требуется только для интерактивного терминала на устройствах, где окно Manager Action не предоставляет стандартный ввод.

## Поведение кнопки Action

Некоторые окна Manager Action в KernelSU Next и SukiSU запускают `action.sh` без интерактивного потока стандартного ввода. Нумерованный цикл с `read` в такой среде не может работать надёжно.

Поэтому F2FS Guardian:

- открывает интерактивное меню только при наличии настоящего TTY;
- в остальных случаях выводит локализованную панель состояния;
- корректно завершается вместо повторения `Invalid selection`;
- предлагает запустить `f2g` из Termux или ADB shell.

## Профили

### Balanced

Политика по умолчанию. Проверка каждые 60 минут, ожидание 24 часа между успешными сессиями, требование 20 минут с выключенным экраном и обычный триггер при заполнении 84% и 256 dirty-сегментах.

### Conservative

Проверка каждые 120 минут, ожидание 48 часов между успешными сессиями, требование 30 минут с выключенным экраном, повышение обычного триггера до заполнения 88% и 512 dirty-сегментов, а также сокращённые лимиты сессий.

### Manual

Отключает автоматическое обслуживание. Мониторинг и поставленные в очередь ручные запросы остаются доступными; обслуживание из очереди по-прежнему подчиняется всем защитным условиям.

## Конфигурация

Постоянная пользовательская конфигурация:

```text
/data/adb/f2fs_guardian/config.conf
```

Заводские значения:

```text
/data/adb/modules/f2fs_guardian/config/default.conf
```

Парсер принимает только фиксированный белый список целочисленных ключей. Неизвестные ключи и недопустимые значения игнорируются. Файл не выполняется как shell-код. Постоянная конфигурация сохраняется при обновлении модуля и удаляется при штатном удалении. `MENU_LANGUAGE=0` выбирает русский язык, `1` — английский, а `2` однократно определяет системную локаль.

## Расшифровка статуса

Распространённые результаты:

| Вывод | Значение |
|---|---|
| `gc_urgent: 0` | Принудительный GC, принадлежащий модулю, не активен |
| `Last run: never` | Ни одна сессия обслуживания ещё не завершилась |
| `no trigger` | Состояние файловой системы не требует обслуживания |
| `waiting: not charging` | Триггер или запрос существует, но требуется зарядка |
| `waiting: battery temperature...` | Безопасное окно заблокировано температурой |
| `Manual request: queued` | Запрос сохраняется до появления безопасных условий или отмены |
| `manual maintenance cancelled` | Запрос в очереди успешно удалён |

Android-команда `ps -A` может показывать демон только как `sh`. Авторитетный PID хранится в:

```text
/data/adb/f2fs_guardian/state/daemon.lock/pid
```

## Журналы и конфиденциальность

```text
/data/adb/f2fs_guardian/logs/guardian.log
/data/adb/f2fs_guardian/state/
```

Размер журнала ограничен, старые записи ротируются. Модуль не передаёт журналы, идентификаторы устройства или телеметрию. Модель безопасности и порядок отправки отчётов описаны в [SECURITY.md](SECURITY.md).

## Конфликты

Не используйте F2FS Guardian одновременно с другим модулем, kernel manager или загрузочным скриптом, который записывает данные в те же узлы F2FS. Модуль обнаруживает заранее установленный ненулевой режим и отказывается запускаться. Если другой инструмент изменит `gc_urgent` во время сессии, F2FS Guardian зафиксирует потерю владения и не станет перезаписывать внешнее значение.

Потенциально конфликтующие категории:

- «оптимизаторы» F2FS GC;
- профили kernel manager, записывающие данные в `/sys/fs/f2fs/*`;
- модули настройки накопителя с собственным фоновым демоном;
- скрипты производителя или кастомного ядра, принудительно устанавливающие режимы GC.

## Удаление и восстановление загрузки

При штатном удалении через root-менеджер запускается `uninstall.sh`, который запрашивает остановку активной сессии модуля, возвращает принадлежащий модулю режим в `0` и удаляет постоянную конфигурацию, состояние и журналы.

Если Android не загружается достаточно далеко для открытия root-менеджера, отключите модуль из доступной root-оболочки ADB:

```sh
adb shell su -c 'touch /data/adb/modules/f2fs_guardian/disable'
adb reboot
```

Recovery может использовать тот же маркер только при возможности расшифровать и смонтировать `/data`.

## Известные ограничения

- На `ext4` польза не ожидается.
- Исправный и слабо заполненный раздел F2FS может никогда не потребовать сессии.
- F2FS GC создаёт дополнительные внутренние записи; повторяющиеся ручные запросы бесполезны.
- Модуль не исправляет повреждение файловой системы.
- Задержки накопителя, вызванные термальным троттлингом, давлением на RAM, поведением приложений, компиляцией шейдеров или планированием CPU/GPU, находятся вне области действия модуля.
- После обновления прошивки или ядра совместимость необходимо проверить заново.
